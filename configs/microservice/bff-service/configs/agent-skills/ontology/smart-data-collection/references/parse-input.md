# Phase 0a：非 CSV 输入 → Markdown

把图片 / PDF / Word / Excel 等非结构化输入统一解析为 Markdown，作为 Phase 0b（字段语义映射）的输入。

本步骤只负责"把内容捞出来"，**不**做字段对齐、**不**碰 KN object-type、**不**生成 CSV、**不**生成 SQL。

## 1.1 图片文件

| 项 | 值 |
|---|---|
| 支持扩展名 | `.jpg` / `.jpeg` / `.png` / `.bmp` 等常见格式 |
| 调用方式 | 通过 Skill 工具显式调用 `yj-ocr-parser` skill |
| 契约参考 | [skills/yj-ocr-parser/SKILL.md](../../yj-ocr-parser/SKILL.md) |
| 必需环境变量 | `MaaS_model_token`（由 yj-ocr-parser 自己校验） |

### 流程

1. 判断输入是 URL 还是本地路径；URL 先下载到本地临时文件（yj-ocr-parser 内部已处理）。
2. 调用 `yj-ocr-parser`，拿到 `data[]`（含 `page_num` / `text` / `type` / `length`）。
3. 把 `data[]` 还原为 Markdown：
   - 检测到"表格类"片段（行内有规则分隔、列对齐）时，输出 Markdown 表格 `| col1 | col2 | ... |`；
   - 其余文字片段保持自然段。
4. 输出统一格式见下方 **输出契约**。
5. 删除临时文件（如有）。

### 注意

- 不要在本步骤里"清洗"字段名（如把"数量"改成 `quantity`）；保留 OCR 原文，由 0b 步骤交给 object-type 字段定义来做对齐。
- OCR 偶尔会把"金额"误识别为"全额"等近似字，0b 步骤的「歧义」分支会兜住。

## 1.2 文档文件（PDF / Word / Excel）

| 项 | 值 |
|---|---|
| 支持扩展名 | `.pdf` / `.doc` / `.docx` / `.xls` / `.xlsx` |
| 调用方式 | 调用文档解析工具（查看有哪些工具或skill可用） |

### 集成约束（不管最终接什么工具，必须满足以下契约）

1. 输出必须包含 **Markdown 表格**：每张表抽成一段 `| col1 | col2 | ... |`，表头作为第一行；多张表用空行分隔。
2. 必须返回 **每个字段的至少一行样值**（用于 0b 步骤判定类型 / 格式 / 长度），写在「字段样值 JSON」里。
3. 必须保留 **原始字段名**：不归一化、不翻译、不去重；归一化是 0b 的职责。

## 输出契约（0a 步骤的统一产物）

无论输入是图片还是文档，0a 步骤必须返回下面这两块内容，作为 0b 步骤的输入。

### 1) Markdown 正文

```markdown
# <文件名或主题>

## 表 1

| 订单号 | 客户名称 | 商品 | 数量 | 金额 |
|--------|----------|------|------|------|
| O001   | 张三     | 笔记本 | 2  | 12000 |
| O002   | 李四     | 鼠标   | 5  | 250   |

## 备注

（其余自然段文本）
```

### 2) 字段样值 JSON

```json
{
  "source": "orders_2026.xlsx",
  "tables": [
    {
      "name": "表 1",
      "columns": [
        {"name": "订单号", "sample": "O001"},
        {"name": "客户名称", "sample": "张三"},
        {"name": "商品", "sample": "笔记本"},
        {"name": "数量", "sample": "2"},
        {"name": "金额", "sample": "12000"}
      ]
    }
  ]
}
```

字段样值 JSON 中：
- `name` 保留原始字段名（含中文 / 大小写 / 空格）；
- `sample` 取第一条非空数据行的值，类型一律字符串（保留原始字面量，类型推断在 0b）。
